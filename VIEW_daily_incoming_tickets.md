# VIEW: CustomerService.DAILY_Incoming_Total

## 1. Purpose
Provides the **daily net incoming ticket volume** used by the business as the single source of truth for demand.
This view consolidates all incoming workload by combining:

- Incoming tickets directly from customers
- Incoming tickets generated via internal transfers

Spam cases are **excluded by design at source level** and must not be included in this aggregation.

This view represents the **official daily incoming volume** used for:
- Operational reporting
- Capacity planning
- Trend and seasonality analysis
- Quality and performance ratios

---

## 2. Grain (CRITICAL)
One row per:
- Date
- Department
- Language

Each row represents the **total net incoming tickets for a department on a given day**, split by language where applicable.

⚠️ **Why grain matters**
- This is the **lowest business‑approved aggregation level** for incoming volume.
- All higher‑level metrics (weekly, monthly, forecast) rely on this daily grain.
- Any duplication or mis‑aggregation here propagates to:
  - Capacity models
  - Reopen ratios
  - Productivity denominators

---

## 3. Functional Definition (Business Logic)

Daily incoming tickets are defined as:
Total Incoming = Incoming tickets from customers
Incoming tickets generated by transfers

### 3.1 Incoming from customers
Sourced from `CustomerService.incoming_tickets`:
- Represents tickets created directly by customers
- Aggregated by date, department and language

### 3.2 Incoming from transfers
Sourced from `CustomerService.agent_transfer`:
- Represents tickets transferred **into** a department
- Only counts transfers where:
  - `new_department ≠ department_id`
  - Both departments are defined
- Language does not apply and is set to `N/A`

### 3.3 Spam exclusion
Spam cases are **not included** in this view.
Spam handling is assumed to be managed upstream and/or via exclusion logic.

## 4. Physical View Definition

```sql
ALTER VIEW CustomerService.DAILY_Incoming_Total
AS
WITH incoming AS (
    SELECT
        it.[Date]                         AS [Date],
        it.department_id                  AS department_id,
        it.language                       AS language,
        SUM(COALESCE(it.ticket_total, 0)) AS incoming_from_customers,
        CAST(0 AS int)                    AS incoming_from_transfers
    FROM CustomerService.incoming_tickets AS it
    WHERE it.[Date] IS NOT NULL
    GROUP BY it.[Date], it.department_id, it.language
),
transfers_in AS (
    SELECT
        at.[Date]                  AS [Date],
        at.new_department          AS department_id,
        CAST(NULL AS nvarchar(50)) AS language,
        CAST(0 AS int)             AS incoming_from_customers,
        COUNT(*)                   AS incoming_from_transfers
    FROM CustomerService.agent_transfer AS at
    WHERE at.[Date] IS NOT NULL
      AND at.new_department IS NOT NULL
      AND at.department_id IS NOT NULL
      AND at.new_department <> at.department_id
    GROUP BY at.[Date], at.new_department
),
unioned AS (
    SELECT * FROM incoming
    UNION ALL
    SELECT * FROM transfers_in
),
aggregated AS (
    SELECT
        u.[Date],
        u.department_id,
        COALESCE(u.language, 'N/A') AS language,
        SUM(u.incoming_from_customers) AS incoming_from_customers,
        SUM(u.incoming_from_transfers) AS incoming_from_transfers,
        SUM(u.incoming_from_customers) + SUM(u.incoming_from_transfers) AS total_incoming
    FROM unioned AS u
    GROUP BY u.[Date], u.department_id, u.language
)
SELECT
    a.[Date],
    a.department_id,
    d.[Week_number],
    d.[Month_number],
    d.[Week_Year] AS [Year],
    a.language,
    a.incoming_from_customers,
    a.incoming_from_transfers,
    a.total_incoming,
    COALESCE(dep.[vertical], 'N/A') AS vertical
FROM aggregated AS a
LEFT JOIN CustomerService.[date] AS d
    ON d.[Date] = a.[Date]
LEFT JOIN CustomerService.[department] AS dep
    ON dep.department_id = a.department_id
WHERE dep.[vertical] IN (
    'Hospitality',
    'Payments',
    'Partners',
    'Onboarding',
    'Finance'
);
```
## 5. Output Column Definitions

| Column name | Type | Description |
|------------|------|-------------|
| Date | date | Calendar date of the incoming aggregation. |
| department_id | int | Department receiving the incoming workload. |
| Week_number | int | Week number from the date dimension. |
| Month_number | int | Month number from the date dimension. |
| Year | string/int | Business year label (Week‑Year logic). |
| language | varchar | Ticket language (`N/A` for transfer‑based incoming). |
| incoming_from_customers | int | Tickets created directly by customers. |
| incoming_from_transfers | int | Tickets transferred into the department. |
| total_incoming | int | Net daily incoming tickets (customers + transfers). |
| vertical | varchar | Business vertical associated with the department. |

---

## 6. Business Rules & Assumptions

- Transfers **increase** incoming volume for the receiving department.
- Self‑transfers (same department) are excluded.
- Language only applies to customer‑created tickets.
- This view represents the **official business definition** of incoming volume.
- Aggregation is performed at daily level only.

---

## 7. Data Quality Considerations

- Relies on correctness of `incoming_tickets` and `agent_transfer`.
- Incorrect department mapping in transfers will distort totals.
- Spam cases must be excluded upstream; no explicit spam join exists here.
- Date is treated as calendar date; no timezone logic is applied.

## 9. Migration Notes (Azure → Snowflake)

✅ **Recommended approach**
- Rebuild this logic as a semantic or mart‑level model.
- Preserve the same business logic for incoming definition.
- Validate daily totals against Azure before aggregating further.

⚠️ **Critical validation**
- Ensure that transfers are counted **once and only once**.
- Ensure language logic (`N/A`) is preserved.

## 10. Known Gaps / Technical Debt

- No explicit flag for **newly boarded customers**
- No explicit flag for **premium customers**
- Spam exclusion logic not explicitly documented in the model
- Language standardization not enforced
- No audit or ingestion metadata
